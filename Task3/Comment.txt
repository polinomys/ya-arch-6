Исходя из имеющегося взаимодействия представляется целесообразным применить подход Event-Driven архитектуры (выборочно).

А именно, отказаться от использовании REST API во взаимодействии между core-app, ins-comp-settlement с сервисом ins-product-aggregator с целью получения данных от страховых компаний. Использовать для этих целей брокер (или очередь) сообщений.
Соответственно, сервисы core-app и ins-comp-settlement в соответствии со своими временными периодами будут получать данные из брокера, которые туда будет складывать сервис ins-product-aggregator.
Это позволит нивелировать дополнительные задержки и сбои в случае увеличения количества страховых компаний, удалит избыточную связность и позволит, при необходимости, скорректировать интервалы запросов данных из страховых компаний.

В тоже время, во взаимодействии core-app с сервисом ins-comp-settlement раз в сутки для получения всех оформленных за день страховок, предлагается сохранить REST API взаимодействие.
Так как запрос делается 1 раз в сутки в период низкой нагрузки.

Предлагаемый вариант изображен на схеме.
